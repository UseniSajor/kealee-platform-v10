# Railway Environment Setup Guide

## Overview

This guide ensures proper isolation between **staging** and **production** environments in Railway, with separate databases and environment variables.

## Architecture

```
Railway Project: graceful-surprise
├── Environment: staging
│   ├── Service: kealee-platform-v10 (staging)
│   │   └── DATABASE_URL → staging-postgres.internal
│   └── Service: staging-postgres
│       └── DATABASE_URL (generated by Railway)
│
└── Environment: production
    ├── Service: kealee-platform-v10 (production)
    │   └── DATABASE_URL → production-postgres.internal
    └── Service: production-postgres
        └── DATABASE_URL (generated by Railway)
```

## Critical Rules

### ❌ DO NOT

- ❌ Share the same Postgres service between staging and production
- ❌ Reuse the same DATABASE_URL across environments
- ❌ Point staging at production data
- ❌ Run `prisma migrate dev` in production (use `prisma migrate deploy`)

### ✅ DO

- ✅ Use separate Postgres services for each environment
- ✅ Use different DATABASE_URL values for each environment
- ✅ Use Railway internal DNS (e.g., `production-postgres.internal`)
- ✅ Run `prisma migrate deploy` in production
- ✅ Run `prisma migrate dev` only in staging/local

## Step-by-Step Setup

### 1. Verify Service Structure

In Railway dashboard, confirm you have:

**Staging Environment:**
- ✅ `kealee-platform-v10` service (or `api-staging`)
- ✅ `staging-postgres` service

**Production Environment:**
- ✅ `kealee-platform-v10` service
- ✅ `production-postgres` service

### 2. Configure DATABASE_URL for Staging

1. Go to **Railway Dashboard** → Your Project → **Staging** environment
2. Click on `kealee-platform-v10` (or `api-staging`) service
3. Go to **Variables** tab
4. Add/Update `DATABASE_URL`:
   - Click **"New Variable"** or edit existing
   - **Name:** `DATABASE_URL`
   - **Value:** Use the **internal** connection string from `staging-postgres`
   - Format: `postgresql://user:password@staging-postgres.internal:5432/railway?schema=public`
   - **Important:** Use `.internal` hostname (not public hostname)

5. **How to get the internal connection string:**
   - Go to `staging-postgres` service
   - Click **Variables** tab
   - Copy the `DATABASE_URL` value
   - Replace the hostname with `staging-postgres.internal`
   - Example: If Railway gives you `postgresql://postgres:password@containers-us-west-123.railway.app:5432/railway`, change it to `postgresql://postgres:password@staging-postgres.internal:5432/railway`

### 3. Configure DATABASE_URL for Production

1. Go to **Railway Dashboard** → Your Project → **Production** environment
2. Click on `kealee-platform-v10` service
3. Go to **Variables** tab
4. Add/Update `DATABASE_URL`:
   - Click **"New Variable"** or edit existing
   - **Name:** `DATABASE_URL`
   - **Value:** Use the **internal** connection string from `production-postgres`
   - Format: `postgresql://user:password@production-postgres.internal:5432/railway?schema=public`
   - **Important:** Use `.internal` hostname (not public hostname)

5. **How to get the internal connection string:**
   - Go to `production-postgres` service
   - Click **Variables** tab
   - Copy the `DATABASE_URL` value
   - Replace the hostname with `production-postgres.internal`

### 4. Verify DATABASE_URL Isolation

Run the verification script:

```bash
# From project root
bash scripts/verify-railway-env-isolation.sh
```

Or manually check:

1. **Staging DATABASE_URL** should contain: `staging-postgres.internal`
2. **Production DATABASE_URL** should contain: `production-postgres.internal`
3. They should be **different** values

### 5. Configure Prisma Migrations

#### For Staging

Staging can use `prisma migrate dev` (development mode):

The `railway.json` is already configured to run migrations during deployment.

#### For Production

Production **must** use `prisma migrate deploy` (production mode):

The `railway.json` is configured to run `prisma migrate deploy` before starting the service.

## Railway Configuration

### railway.json (Root)

The root `railway.json` uses Dockerfile for deployment. Prisma migrations are handled in the Dockerfile.

### services/api/railway.json

This configuration handles migrations automatically:

```json
{
  "build": {
    "builder": "NIXPACKS",
    "buildCommand": "pnpm install && pnpm build"
  },
  "deploy": {
    "startCommand": "cd packages/database && pnpm db:migrate:deploy && cd ../.. && pnpm start",
    "healthcheckPath": "/health",
    "healthcheckTimeout": 100,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 10
  }
}
```

**Note:** The `startCommand` runs `prisma migrate deploy` before starting the service, which is safe for production.

## Common Issues and Solutions

### Issue: Deployment Failures

**Symptom:** Deployments fail with Prisma errors

**Causes:**
1. DATABASE_URL not set or incorrect
2. DATABASE_URL points to wrong environment
3. Prisma migrations failing

**Solution:**
1. Verify DATABASE_URL in Railway dashboard
2. Ensure DATABASE_URL uses `.internal` hostname
3. Check Railway logs for specific Prisma errors
4. Run `prisma migrate status` locally with the DATABASE_URL to verify

### Issue: Staging Affecting Production Data

**Symptom:** Changes in staging appear in production

**Cause:** DATABASE_URL is the same for both environments

**Solution:**
1. Verify DATABASE_URL values are different
2. Check that staging uses `staging-postgres.internal`
3. Check that production uses `production-postgres.internal`
4. Run verification script: `bash scripts/verify-railway-env-isolation.sh`

### Issue: Migration Conflicts

**Symptom:** Migrations fail with "already applied" or "out of sync" errors

**Causes:**
1. Running `prisma migrate dev` in production
2. Migration history mismatch between environments

**Solution:**
1. **Never** run `prisma migrate dev` in production
2. Always use `prisma migrate deploy` in production
3. Ensure migration files are committed to git
4. Check `_prisma_migrations` table in database

### Issue: Connection Refused

**Symptom:** Cannot connect to database

**Causes:**
1. Using public hostname instead of `.internal`
2. Database service not running
3. Wrong port or credentials

**Solution:**
1. Use `.internal` hostname for Railway internal connections
2. Verify database service is online in Railway dashboard
3. Check DATABASE_URL format matches Railway's generated format

## Best Practices

1. **Always verify DATABASE_URL** before deploying
2. **Use internal hostnames** (`.internal`) for Railway service-to-service communication
3. **Never share DATABASE_URL** between environments
4. **Use `prisma migrate deploy`** in production, never `prisma migrate dev`
5. **Test migrations in staging** before applying to production
6. **Monitor deployment logs** for Prisma errors
7. **Backup databases** before running migrations in production
